# Evaluation datasets

## `qa_pairs.jsonl`

Each line is one evaluation question:

```json
{
  "id": "q1",
  "question": "How many days do I have to request a refund?",
  "expected_answer": "...",
  "source_doc": "refund-policy.md",
  "source_keyword": "30 days of purchase"
}
```

### Why `source_doc` + `source_keyword` instead of literal chunk IDs?

A document's `doc_id` is a SHA-256 hash of its **resolved absolute filesystem
path** (see `documind_core.ingestion.pipeline._doc_id_for_path`). That path
differs between machines and checkouts — `/home/alice/documind/...` on one
laptop, `/Users/bob/repo/...` on another, a different path again in CI. If
this file hardcoded literal `chunk_id` values like
`"81d26f253b71a61084d1bae5:0"`, every one of them would be wrong the moment
someone else clones the repo and runs the seed script from a different path.

Instead, each question names the **source file** (`source_doc`, matched
against `Document.source_path`) and a short **exact phrase** that only
appears in the one chunk that actually answers the question
(`source_keyword`). `retrieval_eval.py` resolves this to the real,
currently-valid `chunk_id`(s) at evaluation time by querying the live
SQLite `Chunk` table — so the ground truth is always correct for
*whatever* environment the eval is run in, without needing to be
regenerated by hand after every re-ingest.

### Corpus

The four documents in `corpus/` are short on purpose (a few short sections
each) so that `chunk_document()` produces exactly one or two chunks per
document — this keeps the ground-truth keyword match unambiguous. If you
expand the corpus with longer documents, make sure each `source_keyword`
still appears in only one chunk, or `retrieval_eval.py` will treat all
matching chunks as valid ground truth (which is handled correctly, just
worth knowing).
